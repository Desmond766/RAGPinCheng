01
Phase One · Foundations

Scope the Problem & Pick an Architecture

Before any code, decide whether RAG is even the right tool, and which RAG flavor fits. Skipping this step is the single biggest cause of expensive rebuilds.

Decision tree · at a glance
Knowledge is missing or stale, behavior is fine
→
RAG
Knowledge is fine, output format or tone is wrong
→
Fine-tuning, not RAG
Simple, single-hop Q&A over a known corpus
→
Naive RAG, optimize later
Multi-hop reasoning, entity-heavy domain
→
Graph RAG or hybrid graph+vector
Compound queries spanning docs + APIs + DBs
→
Agentic RAG with tools
Figures, charts, diagrams carry meaning
→
Multimodal RAG
Multi-tenant SaaS or multiple feature lanes
→
Modular RAG as the chassis
Considerations & Decisions
Is RAG actually the right approach?
RAG addresses missing or stale knowledge. It does not fix tone, format, or behavior — that's fine-tuning territory. Test prompt engineering first; if the model lacks specific or fresh facts, add RAG; reach for fine-tuning only when you need to change how the model behaves, not what it knows.
  • Prompt engineeringCheap, fast, no infra. Use when the model already has the knowledge.
  • RAGUse when knowledge is domain-specific, private, or changes over time.
  • Fine-tuningUse to change behavior, format, or style — not to inject facts.
  • Hybrid (RAG + FT)Production systems often need both: FT for format, RAG for facts.
Which RAG architecture pattern?
The choice is driven by query complexity, latency budget, and whether your data has rich relationships.
  • Naive RAGSingle retrieval → single generation. Good for FAQ-style, single-hop questions.
  • Advanced RAGAdds pre-retrieval (query rewriting) and post-retrieval (reranking, compression).
  • Modular RAGPluggable components, routers, conditional paths. Best for multi-tenant production.
  • Graph RAGKnowledge graph as the retrieval substrate. Use for multi-hop reasoning, entity-heavy domains.
  • Agentic RAGLLM decides when/what to retrieve, can call tools. Highest flexibility, highest cost & latency.
  • Multimodal RAGRetrieves across text + images (and audio). Needed when source docs have meaningful figures, charts, diagrams.
How to combine

These aren't mutually exclusive — they're layers. Modular is the chassis; Advanced techniques bolt on; Graph/Multimodal/Agentic are domain extensions.

Advanced RAG on a Modular chassis
Modular routing → Advanced (rewrite + rerank) → generation
The production default. Modular gives you swappable components; Advanced gives you the quality lifts. Almost every mature system runs this.
Hybrid Graph + Vector RAG
Router → {Graph retrieval | Vector retrieval} → fusion → rerank → gen
Microsoft GraphRAG and most production KG systems do this. Vector for fuzzy/descriptive questions; graph for entity-relationship/multi-hop. Don't pick one — route per query.
Agentic + RAG-as-tool
Agent (planner) → calls RAG as one of N tools → synthesizes
When users ask compound questions spanning docs + APIs + databases. The agent decides whether to retrieve at all and from which source.
Multimodal + Text RAG
Parallel retrieval over text index + image index → unified rerank
Technical manuals, slides, scientific papers. Run two retrievers, merge by score, let a vision-capable LLM consume both.
Avoid stacking Full Agentic + Graph + Multimodal at once before you have Naive RAG baseline metrics. Each layer adds 200–800ms and a failure mode. Ship the simplest version that beats your baseline, then layer.
What are your non-functional requirements?
Lock these in early — they constrain every later decision (embedding model size, index type, infra).
  • Latency budget<500ms (real-time chat), <3s (Q&A), batch (offline analytics).
  • Corpus size<10k docs (flat index OK), 10k–10M (ANN required), >10M (sharded/distributed).
  • Freshness SLAReal-time (streaming ingest), hourly, daily, ad-hoc.
  • CompliancePII handling, data residency, on-prem vs cloud, audit logging.
  • Cost ceiling$/query target drives model choice (hosted vs self-hosted, embedding dimensions).
Rule of thumb: Start with Naive RAG end-to-end before optimizing any single component. You cannot tune what you cannot measure, and you cannot measure without a baseline.

Phase Two · Data Pipeline

Load, Parse & Clean the Source Documents

Garbage in, garbage out. Every downstream metric is capped by the quality of this step. Treat it as a first-class engineering problem, not a one-liner.

Decision tree · at a glance
Mixed formats (PDF + Word + HTML + MD)
→
Unstructured as base, route specialists for hard cases
Complex digital PDFs (contracts, scientific papers)
→
LlamaParse, Marker, or MinerU
Scanned PDFs or image-based docs
→
PaddleOCR / Tesseract, fall back from text extract
Source is the open web or dynamic sites
→
FireCrawl / Playwright
Tables or figures carry the answer
→
Structural parser (Marker/MinerU) + multimodal-aware pipeline
Regulated domain with PII or access control
→
Add PII redaction + capture ACL metadata at load time
Considerations & Decisions
What document formats must you support?
PDF is the hardest. Tables, scans, multi-column layouts, and embedded images each need different tooling. Audit your real corpus before picking a loader.
  • UnstructuredMulti-format (PDF/Word/HTML/MD), unified API, element classification. Good general default.
  • PyMuPDF4LLMPDF → Markdown with OCR + tables. Open source, GPU-friendly.
  • LlamaParseDeep structural parsing for complex PDFs (contracts, papers). Commercial API.
  • Marker / MinerUHigh-fidelity PDF → Markdown; MinerU bundles LayoutLM + YOLO for multimodal.
  • DoclingModular, enterprise-grade (IBM ecosystem). Strong on reports, contracts.
  • PaddleOCR / TesseractFor scanned PDFs and image-based documents.
  • FireCrawl / PlaywrightWeb pages, dynamic JS sites, online docs.
  • TextLoader / DirectoryLoaderPlain text and bulk-loading mixed folders.
How to combine

Almost every real corpus is heterogeneous. Production loaders are routers, not single tools.

Format-routed pipeline
MIME sniff → {Unstructured for docs | LlamaParse for complex PDFs | FireCrawl for URLs}
Use a router that inspects each input and dispatches to the right parser. Unstructured covers 80%; specialized parsers fix the long tail.
Fast-path with OCR fallback
PyMuPDF text extract → if low text density, fall back to PaddleOCR / MinerU
Mixed digital and scanned PDFs. Avoids running OCR on every document (slow + expensive) while still handling scans correctly.
Two-pass: structure + content
Marker/MinerU for layout & tables → Unstructured for element classification
High-stakes docs (legal, scientific) where you need both clean Markdown and labeled elements (Title vs NarrativeText vs Table).
Avoid stacking Running every loader on every doc and merging — duplicates content, multiplies cost, and creates ambiguous provenance. One loader per doc, chosen by type.
What metadata do you need to capture?
Metadata enables filtering, citation, freshness control, and access policies later. Extract it during loading — backfilling is painful.
  • ProvenanceSource URL/path, file hash, ingest timestamp.
  • StructurePage number, section, heading hierarchy, paragraph index.
  • Content typeTitle, narrative, list item, table, code, formula, image caption.
  • Domain tagsAuthor, department, product, language, version.
  • Access controlTenant ID, ACL group, classification level.
How do you handle non-text content?
Tables, images, and code each have semantics that get destroyed by naive text extraction. Decide per content type.
  • TablesSerialize to Markdown/HTML, or describe with an LLM, or embed as multimodal.
  • ImagesGenerate captions (BLIP, GPT-4V) or use a multimodal embedding model directly.
  • FormulasPreserve as LaTeX/MathML — never flatten to OCR'd characters.
  • Code blocksKeep verbatim with language tags; never chunk through a function body.
What cleaning rules apply?
  • DeduplicationHash-based (exact), MinHash/SimHash (near-duplicate).
  • Boilerplate removalHeaders, footers, page numbers, nav menus, cookie banners.
  • Encoding normalizationUTF-8, Unicode NFC, smart-quote → ASCII as needed.
  • PII redactionPresidio, regex, or custom NER pipelines.
  • Language detectionfastText / langdetect — route to the right embedding model.

Phase Three · Data Pipeline

Chunk the Documents

Chunking decides what can possibly be retrieved. Get this wrong and no retriever, reranker, or LLM can save you — the answer simply will not exist as a coherent unit.

Decision tree · at a glance
Well-structured docs (markdown, technical manuals, wikis)
→
Structural splitter, fall through to recursive inside sections
Unstructured prose (reports, articles, transcripts)
→
Recursive character splitter (the default)
Mixed content (prose + tables + code in same doc)
→
Route per element type; tables and code as their own units
Answers need broader context than a small chunk holds
→
Parent–child (small-to-big) retrieval
Recall is your bottleneck and you have budget for LLM enrichment
→
Add contextual retrieval or hypothetical-questions enrichment
Tight budget, no enrichment — just need it working
→
Recursive split + prepend doc title and section headers
Considerations & Decisions
Which chunking strategy fits your content?
Pick by content structure, not by what's easiest. Long-form narrative, tables, and code each demand different splits.
  • Fixed-sizeN tokens per chunk. Simple, fast, often suboptimal.
  • Recursive characterSplits on a priority list (paragraph → sentence → word). The pragmatic default.
  • Sentence-basedUses NLTK/spaCy boundaries. Preserves semantic units.
  • Structural / markdown-awareSplits on headings, code fences, list items. Best for technical docs.
  • Semantic chunkingEmbed sentences, split where similarity drops. Quality-up, cost-up.
  • Agentic / LLM-drivenLLM proposes split points. Highest quality, highest cost — use offline.
  • Parent-child / hierarchicalIndex small chunks, retrieve their larger parents for context.
  • Late chunkingEmbed full doc with a long-context model, then split the embedding stream.
How to combine

Strategies stack naturally — outer structure first, then a finer-grained fallback inside each section.

Structural → Recursive (the workhorse)
Split by markdown headings → recursive character split inside each section
Default for technical docs, wikis, manuals. Keeps headings as semantic anchors while bounding chunk size. LangChain's MarkdownHeaderTextSplitter → RecursiveCharacterTextSplitter is exactly this.
Parent–child (small-to-big retrieval)
Chunk into small (256 tok) for embedding → store pointer to large parent (1500 tok) → retrieve small, return parent to LLM
Best precision/context trade-off in practice. Used by LlamaIndex's auto-merging retriever and LangChain's parent-document retriever.
Per-content-type routing
Detect element type → {prose: recursive | tables: row-as-chunk | code: function-as-chunk}
Mixed-content corpora. Never use one chunker for everything; tables and code have natural unit boundaries that prose doesn't.
Semantic + size cap
Semantic splitter to find breakpoints → enforce min/max token bounds
When you want semantic coherence but can't tolerate runaway chunks. Caps prevent semantic chunker from producing 5k-token monsters.
Avoid stacking Agentic + semantic + late chunking on the same pass — three LLM/embedder calls per chunk multiplies ingestion cost 10–50×. Use agentic chunking only for high-value golden corpora.
What chunk size & overlap?
Size trades context against precision. Overlap reduces boundary loss. Both must be tuned empirically — there is no universal best.
  • Small (128–256 tok)High precision, low context. Good for fact lookup.
  • Medium (512–1024 tok)Common default. Balances recall and context.
  • Large (1500–4000 tok)More context, lower precision, more noise in the prompt.
  • Overlap 10–20%Standard. Mitigates info split across boundaries.
  • Zero overlapOnly safe with structural chunking on well-formed docs.
Should you enrich each chunk?
  • Contextual headersPrepend doc title / section path to each chunk text.
  • LLM-generated summariesOne-line summary per chunk, embed both.
  • Hypothetical questionsGenerate questions the chunk answers, index those.
  • Anthropic contextual retrievalLLM writes a short context per chunk explaining its place in the doc.
How to combine

Enrichments live in different "fields" — index multiple representations per chunk and retrieve over the best one (or all).

Multi-vector index (proven)
For each chunk → embed: [raw text, summary, generated questions] → index all three, retrieve any → return raw text
LangChain's MultiVectorRetriever. Big recall lift because user queries match different representations differently — short queries hit questions, descriptive queries hit summaries.
Contextual headers + contextual retrieval
Prepend section path → LLM-generated 50–100 token context → then embed
Anthropic's published recipe; reported ~35–49% reduction in retrieval failures. Pair with BM25 (Step 6) for the full effect.
Lightweight: headers only
Prepend doc title + section hierarchy to each chunk before embedding
Cheapest enrichment, no extra LLM calls. Should be your baseline before any other technique.
Avoid stacking Generating summaries and hypothetical questions and contextual retrieval per chunk — 3 LLM calls × N chunks gets expensive fast. Start with one, measure, add only if eval improves.

Phase Four · Indexing

Choose & Apply the Embedding Model

The embedding model defines the semantic space your retriever searches. Wrong choice = re-embedding the entire corpus later, which is expensive and slow.

Decision tree · at a glance
English-only, low ops appetite, budget OK
→
Hosted: Voyage-3, OpenAI text-embedding-3, Cohere embed-v3
English-only, self-host required (privacy, on-prem)
→
BGE-large-en, E5-large, GTE-large
Multilingual or mixed-language corpus
→
BGE-M3 (also gives sparse vectors for free) or multilingual-e5
Code search dominant use case
→
Voyage-code or Jina-code
Long passages (>1k tokens) must stay whole
→
Jina v3 (8k), Nomic (8k), or Voyage long-context (32k)
Figures and images are part of the corpus
→
SigLIP, BGE-VL, or Jina-CLIP alongside text embedder
Storage cost dominates at >10M vectors
→
Pick a Matryoshka-capable model, plan to quantize
Domain is far from web text (biomedical, legal jargon)
→
Consider contrastive fine-tuning on synthetic pairs
Considerations & Decisions
Hosted API or self-hosted?
  • Hosted (OpenAI, Cohere, Voyage)Zero ops, strong quality, but $/token, data leaves your network.
  • Self-hosted open sourceBGE, E5, GTE, Nomic, Jina, Stella. Control + privacy, you own the GPUs.
  • Sentence-TransformersFramework for fine-tuning your own. Worth it when domain is far from web text.
Which model family by use case?
  • General EnglishOpenAI text-embedding-3, Cohere embed-v3, Voyage-3, BGE-large.
  • Chinese / multilingualBGE-M3, BGE-zh, multilingual-e5, Jina embeddings v3.
  • CodeCodeBERT, Voyage-code, Jina-code, OpenAI text-embedding-3 (decent).
  • Long contextJina v3 (8k), Nomic embed (8k), Voyage (32k).
  • Multimodal (text + image)CLIP, SigLIP, BGE-VL, OpenCLIP, Jina-CLIP.
  • SparseBM25, SPLADE — for keyword/exact-match recall (used alongside dense, see Step 6).
How to combine

Most production systems run multiple embedders in parallel — one model rarely wins on every query type.

Dense + sparse (the universal pairing)
BGE/OpenAI dense vectors + BM25 (or SPLADE) sparse vectors → hybrid retrieve in Step 6
Sparse catches exact IDs, codes, rare terms; dense catches paraphrases. BGE-M3 produces both in one model call — use it if you want simplicity.
Multilingual + monolingual specialist
Language detect → {English: Voyage-3 / OpenAI | Chinese: BGE-zh | other: multilingual-e5}
Mixed-language corpora. Specialist beats multilingual on its home language; multilingual is the safety net.
Text + image (multimodal)
Text chunks → BGE; images/figures → SigLIP or BGE-VL → store in separate collections, query both
When figures carry meaning (diagrams, charts). Don't try to caption-then-embed unless your captions are reliable; native multimodal beats caption pipelines on visual queries.
Avoid stacking Running 3+ dense models in parallel and ensemble-merging — the cost rarely justifies the gain. Dense + sparse + one reranker (Step 6) beats most dense ensembles.
What dimension & precision?
Higher dims = better quality + more storage + slower search. Matryoshka models let you truncate without retraining.
  • 384 / 512 dimsCheap, fast, fine for small-medium corpora.
  • 768 / 1024 dimsStandard production sweet spot.
  • 1536 / 3072 dimsOpenAI defaults. Strong but storage-heavy.
  • Matryoshka (MRL)Train at high dim, truncate to lower at query time.
  • Quantizationint8 / binary embeddings — 4×–32× storage reduction at small recall cost.
How to combine

Matryoshka and quantization compose — together they shrink storage 30–60× with modest recall loss.

Two-stage: binary search → full-precision rerank
Binary-quantized vectors for ANN search (top-1000) → re-score top-1000 with full float vectors → top-k
Cohere's documented approach. ~32× faster ANN, recall recovered by the rerank stage. Standard for >10M vector corpora.
Matryoshka + int8 quantization
Train/use MRL model → truncate from 3072 → 768 dims → quantize each dim to int8
When you want to keep one model but slash storage. OpenAI text-embedding-3 and Nomic both support MRL natively.
Avoid stacking Aggressive MRL truncation (3072→256) + binary quantization without a rerank stage. The two compress losses compound and recall collapses. Always keep a high-precision rerank stage if you compress aggressively.
Should you fine-tune?
  • Off-the-shelfDefault. Modern open models are strong on most domains.
  • Contrastive fine-tuneUse when domain is far from training (biomedical, legal, internal jargon). Needs query–passage pairs.
  • Synthetic pair generationLLM generates queries from chunks to bootstrap training data.
Benchmark first: Run MTEB / C-MTEB scores against your own labeled query set before committing. Public leaderboards do not predict your corpus.

Phase Five · Indexing

Pick a Vector Store & Index Strategy

The vector DB is the knowledge base. The index controls the recall/latency/cost triangle. These two decisions together set the ceiling on production performance.

Decision tree · at a glance
Prototype or single-node app, <100k vectors
→
Chroma, LanceDB, or FAISS — no separate service needed
Postgres already in stack, <5M vectors
→
pgvector / pgvectorscale — one DB, one ops surface
Need full-text + vector together
→
Elasticsearch / OpenSearch or Weaviate
Want managed SaaS, low ops
→
Pinecone (premium) or hosted Qdrant/Weaviate Cloud
Self-hosted at scale, >10M vectors, strong DX matters
→
Qdrant or Milvus
Billion-scale or distributed-first
→
Milvus (cluster mode) or Vespa
Default index for most workloads
→
HNSW — best recall/latency, RAM-heavy
RAM-constrained at scale
→
HNSW + PQ, or IVF + PQ for the very largest
SSD-bound, >100M vectors
→
DiskANN
Considerations & Decisions
Which vector database?
  • MilvusOpen source, distributed, GPU-supported, rich index types. Strong default at scale.
  • QdrantRust-based, payload filtering, strong DX. Great for medium scale.
  • WeaviateModular, hybrid search built-in, GraphQL API.
  • PineconeFully managed SaaS, no ops, premium pricing.
  • pgvector / pgvectorscalePostgres extension. Use when you already have Postgres and want one DB.
  • Elasticsearch / OpenSearchBest when you also need full-text/BM25 in the same store.
  • Chroma / LanceDB / FAISSLightweight, embedded — for prototyping or single-node apps.
  • VespaHeavyweight, hybrid-native, ML re-ranking. Enterprise scale.
Which ANN index algorithm?
Trade-off: build time vs query latency vs recall vs memory.
  • FLAT (brute force)100% recall, slow. Fine under ~100k vectors.
  • HNSWGraph-based. Excellent recall/latency, high RAM. Most common production choice.
  • IVF / IVF_FLAT / IVF_PQCluster-then-search. Smaller memory footprint, lower recall.
  • PQ / SQ quantizationCompress vectors. Combine with HNSW or IVF for huge corpora.
  • DiskANNSSD-resident index — for >100M vectors on commodity hardware.
  • SCANNGoogle's option, strong on high-recall regimes.
  • Annoy (tree-based)Read-only, mmap-friendly. Used by Spotify; less common today.
How to combine

Index algorithms compose at the algorithmic level — quantization layers under graph or cluster structures.

HNSW + PQ (memory-bounded)
HNSW graph navigation + Product Quantization on the stored vectors
Standard Milvus / FAISS recipe when corpus exceeds RAM. Keeps HNSW speed, ~8–16× memory reduction, ~5–10% recall hit.
IVF + PQ (largest corpora)
IVF clustering for coarse search + PQ compression in each cluster
Billion-scale. Lower recall than HNSW+PQ but much smaller memory and faster builds. FAISS's IVFPQ is the workhorse here.
Two-stage: coarse + rerank in-DB
ANN retrieval (compressed) → re-score top-N with full-precision vectors before returning
Recovers most of the recall lost to compression. Milvus, Qdrant, and Weaviate all support this natively.
Avoid stacking HNSW + IVF in series — they're alternatives, not complements. Pick one based on memory budget and corpus size.
Which similarity metric?
Must match how the embedding model was trained. Wrong metric silently destroys recall.
  • CosineDefault for most modern text embedders.
  • Dot productRequired by some models (e.g. some OpenAI variants); equivalent to cosine if normalized.
  • L2 / EuclideanLess common for text; used in some image embeddings.
  • HammingFor binary embeddings.
Index tuning knobs?
  • HNSW: M, efConstructionBuild-time graph density. Higher = better recall, slower build.
  • HNSW: efSearchQuery-time exploration. Higher = better recall, higher latency.
  • IVF: nlist, nprobeNumber of clusters and probed clusters per query.
  • Payload filteringPre-filter (filter then search) vs post-filter (search then filter). Choose per query pattern.
  • Sharding & replicationFor throughput, HA, and corpora that don't fit one node.

Phase Six · Retrieval

Design the Retrieval Layer

Single-vector top-k is a starting point, not a destination. Production retrieval almost always combines techniques: better queries in, better ranking out.

Decision tree · at a glance
Most natural-language Q&A
→
Hybrid (dense + BM25) with RRF fusion + cross-encoder rerank
Queries often contain IDs, codes, exact terms
→
Lean sparse-heavy in the hybrid weighting
Queries are paraphrase-heavy, vague
→
Lean dense-heavy; consider HyDE pre-retrieval
Compound or multi-hop queries
→
Query decomposition + per-sub-query retrieval
Answer might live in a relational DB, not docs
→
Router → Text2SQL for structured queries, RAG for the rest
Strong tenant or access-control requirements
→
Pre-filter on metadata before similarity search
Precision matters more than latency
→
Two-stage rerank (fast → strong), or ColBERT in place of rerank
Token budget is tight relative to retrieved chunks
→
Add contextual compression after rerank
Considerations & Decisions
Which retrieval modes?
  • Dense (vector)Semantic matches, paraphrase-robust. Weak on rare terms, IDs, exact codes.
  • Sparse (BM25 / SPLADE)Keyword-precise. Strong on names, codes, technical terms.
  • Hybrid (dense + sparse)Production default. Fuse with RRF (Reciprocal Rank Fusion) or weighted scores.
  • ColBERT / late-interactionToken-level matching, higher accuracy, higher cost.
  • Metadata filteringHard filters on tenant, date, doc type, language before similarity.
  • Graph traversalFor Graph RAG: Cypher/Gremlin queries over a KG.
  • Text2SQLWhen the answer lives in a relational DB, not a doc.
How to combine

Retrieval modes are nearly always combined. The question isn't "which one" but "in what order."

Filter → hybrid → rerank (production default)
Metadata filter (tenant, date, doc type) → dense + BM25 parallel → RRF fuse → cross-encoder rerank
The most common production stack. Filter cuts the search space; hybrid maximizes recall; rerank maximizes precision. Used by virtually every mature SaaS RAG product.
Hybrid with weighted fusion
α·dense_score + (1-α)·sparse_score, α tuned per query type
When RRF (rank-only) loses score signal you need. Typical α: 0.5–0.7 for natural language, 0.2–0.4 for keyword-heavy queries. Weaviate, Elasticsearch, and Vespa all support this.
Graph + vector (Graph RAG)
Entity extraction from query → graph traversal for related entities → vector retrieval over docs about those entities → merge
Multi-hop questions like "what suppliers of company X were acquired by Y in 2023." Graph finds the entities; vector retrieves the prose.
Router → mode (Text2SQL or RAG)
LLM classifier inspects query → {numeric/aggregation: Text2SQL | descriptive: hybrid RAG}
Mixed corpora with structured + unstructured data. "How many open tickets does team X have?" should never hit a vector index.
Avoid stacking ColBERT + cross-encoder rerank on the same stage — both are precision tools, you pay twice for similar lift. Pick one; ColBERT generally replaces the rerank stage.
Pre-retrieval: how do you transform the query?
  • Query rewritingLLM rephrases for clarity, expands abbreviations, fixes typos.
  • Query expansionAdd synonyms, related terms (e.g. multi-query retriever).
  • HyDELLM writes a hypothetical answer; embed that instead of the question.
  • Query decompositionBreak compound questions into sub-queries, retrieve per sub-query.
  • Step-back promptingGenerate a more general query, retrieve broadly, then narrow.
  • RoutingClassifier picks which index/tool/source to query.
How to combine

You almost never pick just one. These techniques live at different layers — routing (where to look), reformulation (what to ask), multiplication (one query → many) — and stack naturally.

Routing → rewriting → multi-query (the pragmatic default)
Router picks index → LLM rewrites query → generate 3–5 paraphrases → retrieve for each → RRF fuse
General-purpose production setup. Routing handles multi-source corpora; rewriting cleans typos and pronouns; multi-query lifts recall. Implemented in LangChain's MultiQueryRetriever.
Decomposition + per-sub-query rewrite (multi-hop)
LLM decomposes compound question → rewrite each sub-query → parallel retrieve → synthesize
Best for "compare X and Y" or "trace the chain from A to D" questions. LlamaIndex's SubQuestionQueryEngine is this pattern.
Step-back + HyDE (vague/short queries)
Step-back generates a more general query → HyDE writes a hypothetical answer → embed the hypothetical → retrieve
Two-word queries, jargon-heavy queries, or queries that don't share vocabulary with the corpus. Step-back broadens; HyDE bridges the query–passage vocabulary gap.
Dual-query for hybrid retrieval
Rewritten query → dense retriever (semantic) + original raw query → BM25 (keyword)
BM25 wants the original keywords; the embedder wants clean paraphrases. Don't feed both the same string — split them.
Avoid stacking HyDE + decomposition on the same query (overkill — pick whichever the failure mode demands). Multi-query expansion on small corpora (just hits the same chunks). Stacking 4+ transforms before measuring — every layer adds 200–800ms.

Order of adoption: Start with rewriting + routing. Add decomposition if eval shows multi-hop failures. Add HyDE if short/vague queries fail. Add multi-query expansion if recall is your bottleneck after reranking.

Post-retrieval: how do you re-rank & compress?
  • Cross-encoder rerankerBGE-reranker, Cohere Rerank, Jina reranker. Massive precision lift over top-50.
  • LLM-as-rerankerUse a small LLM to score relevance. Slower, sometimes higher quality.
  • RankRAG / RankGPTListwise ranking with an LLM.
  • MMR (Maximal Marginal Relevance)Diversifies results — avoids 5 near-duplicate chunks.
  • Contextual compressionDrop irrelevant sentences inside each chunk before sending to LLM.
  • Parent doc expansionRetrieve small chunks, expand to surrounding context.
How to combine

Post-retrieval is a pipeline: rerank for relevance, diversify for coverage, expand for context, compress for token budget. They run in order.

Rerank → MMR → parent expand (workhorse)
Top-50 from retrieval → cross-encoder rerank → MMR diversify to top-10 → expand each to parent chunk → send to LLM
Standard for production RAG. Rerank lifts precision; MMR prevents 10 near-duplicate chunks; parent expansion restores the context that small-chunk retrieval lost.
Rerank → contextual compression (long context, tight budget)
Cross-encoder rerank → LLM extracts only the relevant sentences from each chunk → concatenate
When you want top-20 chunks worth of recall in a top-5 token budget. LangChain's ContextualCompressionRetriever. Adds an LLM call but cuts prompt tokens substantially.
Two-stage rerank (highest quality)
Top-100 → fast cross-encoder (BGE-reranker-base) → top-20 → strong reranker (Cohere Rerank-3 or LLM-as-judge) → top-5
When precision matters more than latency. Common in enterprise search and legal/medical RAG.
RRF then rerank (hybrid pipeline)
Dense top-50 + sparse top-50 → RRF fuse to top-50 → cross-encoder rerank to top-10
The bridge between Step 6's hybrid retrieval and a quality reranker. RRF normalizes the score scales before the reranker sees them.
Avoid stacking Cross-encoder + LLM-as-reranker in series (redundant — both score relevance). Parent expansion + MMR before reranking (you'll rerank duplicates). Contextual compression without a rerank step (compresses irrelevant chunks too).
How many chunks do you pass downstream?
  • top-k retrieve = 20–100Wide net for the reranker.
  • top-n after rerank = 3–10What actually goes in the prompt.
  • Token budget awarePack chunks until you hit a fraction (e.g. 50%) of the LLM context window.

Phase Seven · Generation

Integrate the LLM & Control Output

Retrieval found the facts. Generation determines whether the answer is correct, faithful, and usable. Treat the prompt and output format as code, with version control and tests.

Decision tree · at a glance
Most production RAG, no special constraints
→
Mid-tier hosted (Haiku, GPT-4o-mini, Gemini Flash)
High-stakes reasoning or nuance
→
Frontier hosted, or cascade mid-tier → frontier on failure
Privacy or on-prem required, decent hardware
→
Llama 3.x 70B, Qwen 2.5, DeepSeek
High volume, cost-sensitive, strong retrieval upstream
→
Open-weights 7B–13B class
Output is consumed by another program
→
JSON mode / tool calling + schema validation + retry
Open-weights model, can't trust native JSON mode
→
Guided decoding (Outlines, XGrammar)
Regulated or customer-facing
→
Full input + output guardrail pipeline, citation verification
Hallucination is the biggest risk
→
Strict refusal policy + faithfulness judge on output
Considerations & Decisions
Which LLM, hosted or local?
  • Frontier hostedClaude, GPT-4-class, Gemini. Highest quality, easy to start.
  • Mid-tier hostedClaude Haiku, GPT-4o-mini, Gemini Flash. Best $/quality for most RAG.
  • Open weights (large)Llama 3.x 70B, Qwen 2.5, DeepSeek. Self-host for privacy / cost at volume.
  • Open weights (small)7B–13B class. Fast, cheap, good with strong retrieval.
  • SpecializedCode models, multilingual models, function-calling tuned variants.
How to combine

Production rarely uses one model for everything. Cascade by cost/quality, route by query type.

Cascade: small → large on failure
Mid-tier model attempts answer → confidence/judge check → escalate to frontier model only if it fails
Cuts cost 60–80% on easy queries while preserving quality on hard ones. Common with Haiku→Sonnet or 4o-mini→4o.
Router by task
Classifier inspects query → {code: code-specialist | extraction: small fast | reasoning: frontier}
Multi-feature products. Specialized models beat general models on their target task at lower cost.
Generator + judge (different models)
Strong model generates answer → smaller, independent model judges faithfulness/citations → retry on fail
Quality-critical paths. The judge being a different model is the point — same model is too sycophantic about its own output.
Avoid stacking Multiple frontier models in ensemble (e.g. averaging GPT-4 + Claude + Gemini outputs) — 3× cost, marginal gain for most RAG. Cascading beats ensembling on cost/quality.
How do you build the prompt?
  • System promptRole, scope, refusal rules, citation requirements, format spec.
  • Context blockRetrieved chunks, separated by clear delimiters (e.g. double newline, XML tags).
  • Citation instructionsRequire inline source IDs; reject answers without them in post-processing.
  • Refusal policy"If the context doesn't contain the answer, say so" — never invent.
  • Few-shot examplesFor tricky formats or domain-specific tone.
Structured output?
  • Free-form textEasiest, hardest to consume programmatically.
  • JSON mode / response_formatNative on OpenAI, Anthropic, etc.
  • Function/tool callingSchema-validated outputs.
  • Guided decodingOutlines, Guidance, JSON Schema constraints, regex grammars.
  • Pydantic / Zod validationPost-parse validation with retry on failure.
How to combine

Structured output is a defense-in-depth problem. The provider's native mode gets you 95%; validation + retry covers the rest.

Native JSON mode + schema validation + retry
Tool/JSON mode with schema → Pydantic validates output → on failure, send error back to LLM with retry instructions
The production standard. Instructor and BAML libraries package this loop cleanly. Reaches ~99.9% well-formed output.
Guided decoding for open-weights
Outlines/XGrammar constrains token sampling to grammar → guarantees parseability
Self-hosted open-weights stacks where you don't trust native JSON mode. Slightly slower, but invalid output becomes structurally impossible.
Avoid stacking Free-form text + post-hoc regex extraction in production. Works in demos, fails at scale. Use schema-constrained output from the start.
Safety & guardrails?
  • Input filtersPrompt injection detection, PII detection, abuse classification.
  • Output filtersFaithfulness check vs context, toxicity, leaked PII.
  • Citation verificationProgrammatically check every claim points to a retrieved chunk.
  • FrameworksNeMo Guardrails, Guardrails AI, LLM Guard.
How to combine

Guardrails belong on both sides of generation. Defense in depth — no single check catches everything.

Input → generation → output (full pipeline)
Input: prompt-injection + PII detection → generation with strict system prompt → output: faithfulness check + citation verification + toxicity
Standard for customer-facing or regulated deployments. Each layer catches a different failure mode.
Citation verification + faithfulness check
Parse claims from answer → check each cited chunk actually supports it (NLI model or LLM judge) → flag/strip unsupported claims
High-stakes domains (legal, medical, finance). Cheap to run, hardest hallucination control you can buy.
Avoid stacking Multiple guardrail frameworks (NeMo + Guardrails AI + LLM Guard) without clear ownership — they overlap and create conflicting rules. Pick one framework, supplement with custom checks.
Streaming & UX?
  • Token streaming (SSE/WS)Lower perceived latency.
  • Citation renderingInline source pills with hover previews.
  • Show-the-thinkingDisplay retrieved sources before final answer to build trust.

Phase Eight · Quality

Evaluate Systematically

You cannot improve what you do not measure. Build evaluation before scaling — every tuning decision needs a ground truth to point at.

Decision tree · at a glance
Cold start, no labeled data
→
LLM-synthesize Q/A pairs → human review the top 100–300
Mature product, real users
→
Golden set (regression) + sampled production logs (drift)
Selecting an embedder or reranker
→
Public benchmarks (MTEB, BEIR) on your domain slice
Want quick, no-ground-truth eval
→
Ragas (faithfulness, context relevance, answer relevance)
Need tracing and debugging UI too
→
Phoenix, LangSmith, or Langfuse
Want evals in your CI pipeline
→
DeepEval (pytest-style) or Promptfoo (config-driven)
Bottleneck unknown — retrieval or generation?
→
Measure retrieval metrics (Recall@k, MRR) before answer quality
Considerations & Decisions
What is your eval dataset?
  • Hand-curated golden set50–500 question/answer pairs with reference contexts. Highest signal.
  • LLM-synthesizedGenerate Q/A pairs from your corpus (Ragas, LlamaIndex). Cheap, noisier.
  • Production logsReal user queries — most realistic, requires labeling.
  • Public benchmarksMS MARCO, BEIR, MTEB, KILT — useful for embedder/reranker selection only.
How to combine

One dataset is never enough. Mature teams maintain three in parallel, each for a different purpose.

Three-tier eval set
Public benchmark (component selection) + golden set (regression) + production logs (drift detection)
Production-grade. Public benchmarks pick the embedder; golden set is your CI gate; logs catch real-world drift the curated sets miss.
Synthetic bootstrap + human curation
LLM synthesizes 500 Q/A pairs from corpus → humans review, edit, reject ~30% → result is the golden set
Cold-start when you have no labeled data. Far faster than writing from scratch; quality beats pure synthetic.
Avoid stacking Treating LLM-synthesized data as ground truth without human review — synthetic Q/A overfits to the same LLM you're trying to evaluate, masking real failures.
Which retrieval metrics?
  • Recall@kDid the right chunk make it into top-k?
  • Precision@k / MAPHow clean is the top-k?
  • MRRWhere in the ranking is the first relevant chunk?
  • nDCGGraded relevance, position-aware.
  • Context Precision / RecallRagas — relevance of retrieved chunks to question and ground truth.
Which generation metrics?
  • Faithfulness / groundednessAre the claims supported by retrieved context? Critical.
  • Answer relevanceDoes the answer address the question?
  • CorrectnessMatch against ground-truth answer (when available).
  • Citation accuracyDo cited sources actually support the claim?
  • Hallucination rateClaims with no support in context.
Which framework?
  • RagasRAG-specific, no-ground-truth options, lightweight.
  • TruLensTriad of metrics (context relevance, groundedness, answer relevance) + tracing.
  • Phoenix (Arize)OpenTelemetry-based tracing, evals, debugging UI.
  • DeepEvalPytest-style assertions on LLM outputs.
  • LangSmith / LangfuseEnd-to-end tracing + eval workflows.
  • PromptfooConfig-driven, CI-friendly evals.
  • Custom LLM-as-judgeAlways validate the judge against human labels.
Eval cadence?
  • Pre-commit / CIRun a smoke set on every change.
  • Pre-releaseFull eval suite + regression check.
  • Online / productionSample live traffic, run async evals, dashboard them.
  • User feedback loopThumbs up/down, flag-to-review pipeline.
How to combine

Eval is continuous, not a phase. Each cadence layer catches a different class of regression.

CI smoke + pre-release full + online sampled (proven cadence)
PR: 20-question smoke set (<2min) → release: 500-question full suite → prod: 1% sampled async eval + dashboard
The standard for serious teams. Smoke catches obvious breaks; full suite catches subtle regressions; online catches drift the eval set doesn't model.
User feedback → eval set growth
User thumbs-down → flag in queue → label weekly → high-quality examples enter golden set
Closes the loop. Production failures become regression tests; the golden set stays representative of real usage.
Avoid stacking Running the full eval suite on every commit — slow CI, ignored failures. Smoke fast, full slow, online sampled.

Phase Nine · Operate

Deploy, Observe, Iterate

Going to production is the start, not the end. RAG systems drift: data changes, queries change, models change. Operate with the same rigor as any data system.

Decision tree · at a glance
Source data is stable (manuals, archives)
→
Scheduled batch reindex (daily or weekly)
Source updates frequently, hour-level freshness OK
→
Incremental upserts via CDC
Real-time freshness required (news, tickets, chat)
→
Streaming ingest + periodic reconciliation backstop
Changing embedding model on a live system
→
Blue/green index swap, no in-place migration
Cost is dominant pain
→
Start with embedding cache + provider prompt cache (safe)
Multi-tenant SaaS
→
Per-tenant namespaces + hard filter on every query + audit log
Need provider failover or rate-limit handling
→
LLM gateway (LiteLLM, Portkey)
Quality regressing over time
→
Online sampled evals + mine thumbs-down into eval set
Considerations & Decisions
How do you serve the system?
  • Orchestration frameworkLangChain, LlamaIndex, Haystack, DSPy, LangGraph, or hand-rolled.
  • Service layerFastAPI, Litestar; or hosted (Vercel, Modal, RunPod).
  • LLM gatewayLiteLLM, Portkey, OpenRouter — provider failover, rate limiting, cost tracking.
  • Async / queueFor long-running or batch jobs (Celery, Temporal, Redis streams).
How is the index kept fresh?
  • Batch reindexScheduled, full rebuild. Simple, expensive at scale.
  • Incremental upsertsCDC from source, doc-level upsert to vector DB.
  • Streaming ingestKafka → embedder → store. For real-time freshness.
  • Versioned indexesBlue/green index swaps for zero-downtime model migrations.
  • TTL / soft deleteExpire stale docs without re-embedding the world.
How to combine

Freshness is a tiered problem. Different doc types need different update frequencies.

Streaming for hot data + batch for cold (tiered freshness)
Kafka CDC → embedder → incremental upserts for hot docs (news, tickets); weekly batch reindex for cold (manuals, archives)
The standard at scale. Real-time updates are expensive — only pay it for data that actually changes.
Blue/green for model migrations + incremental for content
New embedding model → build green index in parallel → cut over at flip → run incremental upserts on green going forward
When you need to change the embedding model without downtime. Blue stays serving until green is fully built and warmed.
Avoid stacking Streaming ingest with no batch backstop — eventually a stream consumer fails silently and the index drifts. Always have a periodic reconciliation job.
What do you monitor?
  • Latency (p50/p95/p99)Per stage: embed, retrieve, rerank, generate.
  • Cost per queryTokens in/out by model, broken down by tenant/feature.
  • Retrieval healthEmpty-result rate, low-score rate, filter-collapse rate.
  • Quality driftRolling Ragas/judge scores on sampled traffic.
  • Data driftNew entities, vocabulary changes, query topic shift.
  • TracingOpenTelemetry, Phoenix, LangSmith — full request lineage.
Caching strategy?
  • Embedding cacheHash(query) → vector. Cheap, huge hit rate on FAQs.
  • Semantic cacheMatch similar past queries, return cached answer (with caution).
  • Retrieval cachequery+filters → chunk IDs.
  • Prompt cachingAnthropic / OpenAI cached system+context for repeated prefixes.
How to combine

Caching layers stack — each one short-circuits a different stage of the pipeline. Apply them in order of cheapest-hit-first.

Four-layer cache stack (full pipeline)
L1 semantic cache (exact answer) → L2 embedding cache (skip embedder) → L3 retrieval cache (skip vector DB) → L4 prompt cache (skip LLM input tokens)
Each layer hits at a different rate. Semantic catches the top FAQ; embedding cache catches repeated queries; retrieval cache catches similar queries with same filters; prompt cache catches stable system prompts and contexts.
Conservative: embedding + prompt only
Hash(query) → vector cache, plus provider-side prompt caching on system prompt
Safe default. No risk of serving stale or wrong answers (semantic cache can do that). 30–50% cost reduction with zero correctness risk.
Avoid stacking Semantic caching on personalized or time-sensitive answers — "what's the latest update on ticket #1234" should never be served from another user's cache. Scope semantic caches by tenant + freshness boundary.
Security & access control?
  • Tenant isolationPer-tenant namespace, hard filter on every query.
  • Row/doc-level ACLsUser permissions encoded as metadata filters.
  • Prompt injection defenseTreat retrieved content as untrusted input; sanitize, sandbox, never let it override instructions.
  • Audit loggingWho asked what, what was retrieved, what was answered.
  • Rate limits & quotasPer-user, per-tenant, per-model.
How do you iterate?
  • A/B testingShadow traffic, side-by-side eval, feature flags.
  • Error miningCluster failed queries, label, feed back into eval set.
  • Component swapsHold pipeline constant, swap one piece (embedder, reranker), measure.
  • Cost regression testsCatch token-cost blowups before they ship.
Closing note: Production RAG is 20% retrieval cleverness and 80% data discipline, evaluation discipline, and operational discipline. The teams that win are not the ones with the fanciest reranker — they are the ones with the cleanest pipeline and the tightest feedback loop.
Appendix A · Worked Example

Internal Knowledge Base & Q&A over Docs

One end-to-end pass through all nine steps, applied to a realistic, common scenario. The choices below are not the only valid ones — they're a concrete, defensible default stack that shows how decisions hang together.

Scenario

A mid-size company (~2,000 employees) wants an internal Q&A assistant over its policy documents, engineering wikis, HR handbooks, and onboarding materials. Employees ask things like "how many days of parental leave do I get?", "what's our refund policy for enterprise customers?", or "how do I provision a new staging environment?" Answers must cite sources, respect departmental access, and stay current as docs change.

Corpus size~50,000 documents (PDF + Confluence + Markdown)
Users~2,000 employees, ~200 concurrent peak
Latency targetp95 < 3 seconds end-to-end
FreshnessUpdates visible within 1 hour
ComplianceOn-prem preferred; some PII; departmental ACLs
BudgetMid-tier — cost-aware but quality matters
01

Scope & Architecture

Knowledge is private and updates often, so RAG is the right call — fine-tuning would be both expensive and wrong (knowledge changes weekly). Queries are mostly single-hop policy questions but some span multiple documents ("compare leave policy in the US vs UK office"), so we plan for Advanced RAG on a Modular chassis — a routing layer in front, so we can later add a Text2SQL branch for HR-system queries without rewriting. We skip Graph and Agentic RAG for v1 — they don't justify their complexity yet.

ApproachRAG (not fine-tuning)
PatternAdvanced RAG, Modular chassis
Latency budgetp95 < 3s, gives ~1.5s for retrieval+rerank
Corpus classMedium (50k docs → ~500k chunks)
ComplianceOn-prem self-hosted; ACL-aware retrieval
02

Data Loading & Parsing

The corpus is heterogeneous: PDFs (policies, contracts), Confluence pages (wikis), and a Git repo of Markdown (engineering docs). We use Unstructured as the base because it covers all three reasonably and gives a unified element model. For the few complex PDFs (contracts with nested tables), we route to LlamaParse. We capture rich metadata at ingestion — backfilling ACL info would be painful later.

PDF loaderUnstructured, LlamaParse for complex contracts
ConfluenceAtlassian API → HTML → Unstructured
MarkdownNative splitter on headings
Metadatasource URL, doc owner, department, last-modified, ACL group
TablesSerialized to Markdown, kept whole
CleaningDedup by hash, strip Confluence chrome, Presidio for PII tags
03

Chunking

Most docs have clear structure (headings, sections), so we do structural splitting first, then fall through to recursive character splitting inside each section to keep chunks bounded. We use parent–child retrieval — embed small chunks for precision, return their larger parent sections to the LLM for context. We prepend the doc title and section path to each chunk before embedding (cheap, free recall lift).

StrategyStructural → recursive, parent–child
Child chunk~300 tokens, 50 token overlap (for embedding)
Parent chunk~1200 tokens (returned to LLM)
EnrichmentPrepend doc title + section path; skip LLM summaries in v1
Tables & codeKept as single chunks, never split mid-row/mid-function
04

Embeddings

Corpus is mostly English with some Chinese product docs for the APAC office, and we need on-prem. BGE-M3 fits perfectly — multilingual, strong on technical text, and it produces both dense and sparse vectors in one model call, which simplifies the hybrid retrieval stage (Step 6). We skip fine-tuning for v1: BGE-M3 is strong enough out of the box; revisit only if eval shows domain-specific failures.

ModelBGE-M3 (self-hosted on internal GPU)
Dimension1024 (native), no quantization yet
OutputsDense + sparse from same forward pass
Fine-tuningNone in v1; reconsider after first eval pass
05

Vector Store & Index

Self-hosted, ~500k chunks, want strong hybrid support and good DX. Qdrant is the sweet spot — Rust core, excellent payload filtering (critical for ACLs), supports both dense and sparse vectors, and ops is simpler than Milvus at this scale. HNSW is our index — recall and latency both matter more than RAM at 500k vectors. Cosine similarity to match BGE-M3's training.

Vector DBQdrant (self-hosted, single cluster)
IndexHNSW (M=16, efConstruction=200)
MetricCosine
Payload filtersdepartment, ACL group, doc_type, last_modified
ShardingNone yet — single node sufficient
06

Retrieval

The workhorse pipeline: ACL filter → hybrid retrieve → RRF fuse → cross-encoder rerank → parent expansion. The ACL filter is non-negotiable: it runs before similarity so users never see chunks they can't access. Hybrid catches both paraphrase questions ("how much vacation do I get") and ID-style ones ("Policy HR-2024-07"). A lightweight LLM rewrite handles typos and pronouns. Decomposition and HyDE wait for v2 unless eval demands them.

Pre-retrievalLLM rewrite (Haiku, cheap) → routed by query classifier
FilteringACL + department metadata pre-filter
RetrievalDense + sparse (both from BGE-M3) in parallel
FusionRRF, top 50 candidates
RerankBGE-reranker-v2-m3 cross-encoder → top 5
Post-processingExpand to parent chunks, MMR for diversity
07

Generation

Mid-tier hosted LLM is the cost/quality sweet spot, but our on-prem requirement means we self-host. Llama 3.x 70B-Instruct on internal GPUs covers it. The system prompt enforces citations, strict refusal when context is missing, and a faithfulness check on the output before serving. We don't need structured output for chat answers, but we do enforce a "[source: doc_id]" citation pattern that we verify post-hoc.

LLMLlama 3.x 70B-Instruct (self-hosted via vLLM)
PromptRole + retrieved context (XML-delimited) + refusal rule + cite rule
CitationRequired inline [source: doc_id]; post-parse to verify
Refusal"If context doesn't contain the answer, say so" — never invent

08

Evaluation

Cold start, so we bootstrap with LLM-synthesized Q/A pairs reviewed by humans — engineers and HR partners label ~300 questions from real-looking categories. Ragas for the core metrics (faithfulness, context precision/recall, answer relevance), wired into CI via a smoke set. Phoenix gives us tracing so we can debug bad queries by drilling into a single trace. Once live, we sample 1% of production traffic for ongoing eval and feed thumbs-down examples back into the golden set.

DatasetsGolden set (300, human-reviewed) + sampled prod logs
FrameworkRagas for metrics, Phoenix for tracing
Retrieval metricsRecall@10, MRR, context precision
Generation metricsFaithfulness, citation accuracy, answer relevance
CadenceCI smoke (20 Qs) + pre-release full + 1% prod sampling
09

Deployment & Ops

FastAPI service in front of Qdrant + vLLM, orchestrated with LangGraph (modular nodes for routing, retrieve, rerank, generate). Confluence and Git both emit webhooks → small ingest queue → incremental upserts to Qdrant, giving us the 1-hour freshness SLA. Embedding cache short-circuits repeat questions, prompt caching on the system prompt cuts token cost ~30%. Per-user ACL enforcement at retrieve time + audit log for every query. A/B framework runs from day one so we can swap components without guesswork.

ServingFastAPI + LangGraph, vLLM for the 70B model
FreshnessWebhook-driven incremental upserts; nightly reconciliation
MonitoringPer-stage p50/p95 latency, token cost, retrieval-empty rate, drift
CachingEmbedding cache + Anthropic prompt caching on judge model
SecurityACL filter on every query, audit log, prompt-injection sanitizer
IterationShadow traffic A/B, weekly review of thumbs-down clusters
What this example doesn't include — and why: No HyDE, no decomposition, no Graph RAG, no agentic tools, no quantization, no two-stage rerank. Each of those is a real technique covered in the main steps, but adding them in v1 would inflate complexity without proven need. Ship the baseline, measure where it fails, then layer. Every production-grade RAG system starts simpler than its final form.
